Kubernetes Scaling

Event-Driven Autoscaling with KEDA

Challenge: Workloads were either over-provisioned during quiet periods or slow to respond when queue-backed demand spiked, making static scaling rules costly and unreliable across production tenants.

Platform Engineering

Multi-Cluster GKE Platform Modernisation

Scope Multi-project GCP Kubernetes platform
Focus Standardisation, upgrades, governance
Outcome Safer repeatable cluster operations

Context

Kubernetes clusters were running across multiple GCP projects, but provisioning, configuration, and upgrade practices were not consistent everywhere. That made it harder to reason about cluster state, compare environments, review infrastructure changes, and perform upgrades with confidence.

Exactly What I Did

  • Audited existing cluster setup patterns across GCP projects to identify drift, duplicated configuration, and manual provisioning steps.
  • Created reusable Terraform structure for GKE cluster configuration, node pools, project-level inputs, environment variables, and shared platform defaults.
  • Moved cluster changes into a pull-request based workflow so infrastructure updates could be reviewed before being applied.
  • Added GitHub Actions validation for terraform fmt, terraform validate, and policy checks before merge.
  • Used OPA policy checks to catch unsafe or non-standard infrastructure changes early in CI.
  • Defined a blue-green upgrade approach where a new cluster version could be prepared, validated, and cut over to instead of upgrading in place blindly.
  • Documented the operating flow for cluster provisioning, upgrade preparation, cutover, and rollback decision points.

Upgrade Flow

For upgrades, I treated the target cluster as a controlled replacement path rather than a risky one-shot change. The newer cluster could be provisioned with Terraform, checked against the same policy rules, validated with platform workloads, and then used for a planned cutover. This gave the team a cleaner rollback path and reduced the pressure of doing major changes directly on the active cluster.

Technical Decisions

  • Used Terraform as the source of truth so cluster configuration lived in version control instead of individual engineer machines.
  • Kept project and environment differences in inputs, while common platform behavior stayed standardised.
  • Put validation in CI to catch formatting, syntax, and policy problems before apply-time.
  • Used blue-green cluster patterns where upgrade risk was high enough to justify a parallel environment and controlled migration.

How It Changed Day-to-Day Operations

Engineers could review cluster changes through pull requests, understand which settings differed between environments, and follow a repeatable process for upgrades. Platform changes became less dependent on manual memory and more visible through code, CI results, and documented operating steps.

Impact

The platform became more predictable to operate, with stronger IaC controls, lower upgrade risk, and a clearer path for managing clusters consistently across production-focused environments.

Kubernetes Scaling

Event-Driven Autoscaling with KEDA

Scope Multi-tenant Kubernetes workloads
Focus Queue-aware scaling and resilience
Outcome Faster burst response with less idle capacity

Context

Static HPA configuration was not responding well to tenant-specific workload patterns. Queue-backed processing needed scaling logic tied to real demand so workloads could scale out quickly during bursts and scale back down when pressure dropped.

What I Delivered

  • Introduced KEDA ScaledObjects using queue depth and external demand signals as scaling triggers.
  • Integrated KEDA with Kubernetes HPA so consumer workloads scaled with actual event pressure.
  • Replaced static HPA assumptions with dynamic per-tenant scaling strategies.
  • Implemented tenant-specific max-replica limits to control resource allocation under load.
  • Added fallback behavior so workloads stayed resilient when metrics were delayed or unavailable.

Technical Shape

The implementation used Kubernetes autoscaling primitives, KEDA trigger definitions, and workload-level limits. The important trade-off was balancing fast scale-out behavior with guardrails that kept noisy tenants from consuming disproportionate capacity or driving unnecessary idle spend.

Impact

Scaling became more aligned with actual workload pressure, improving responsiveness during bursts, reducing idle compute during quieter periods, and lowering the need for manual capacity tuning.

Cost Optimisation

GKE Scalar – Serverless Cluster Cost Optimisation

Scope Non-production GKE environments
Focus Scheduled capacity and automation
Outcome Lower idle compute spend

Context

Non-production clusters were staying fully provisioned outside working hours. The cost profile did not match actual usage, but the solution still needed to preserve predictable availability when engineers needed the environments.

What I Delivered

  • Built a serverless scheduler using Cloud Scheduler and Cloud Functions.
  • Automated GKE node pool resizing for working-hours and off-hours capacity profiles.
  • Packaged the implementation as a reusable Terraform module for consistent rollout.
  • Kept the operating model lightweight so teams could adopt it without extra platform overhead.

Technical Shape

The architecture used managed GCP services to trigger capacity changes without maintaining a dedicated controller. Terraform handled repeatability, while scheduled serverless execution kept the system simple and low-maintenance.

Impact

Non-production compute waste was reduced while environments remained available during working hours. The module-based approach also made the pattern reusable across additional clusters.
