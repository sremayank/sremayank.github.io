How to Protect Kubernetes Using Namespaces

Namespaces organize cluster resources and provide boundaries for access control, policy, quotas, and operations. They become protective when you attach the right controls to them.

Published June 12, 202617 min readKubernetes

Important: a namespace is not a security boundary by itself. It does not automatically block network traffic, restrict API access, isolate nodes, or prevent privileged Pods. Use namespaces as the place where RBAC, NetworkPolicy, quotas, and admission controls are applied.

What Namespaces Do and Do Not Protect

Most Kubernetes workload resources are namespace-scoped, including Pods, Deployments, Services, ConfigMaps, Secrets, Roles, and PersistentVolumeClaims. Namespaces let teams reuse names and give policies a clear scope.

Some important resources are cluster-scoped, including Nodes, PersistentVolumes, StorageClasses, ClusterRoles, CustomResourceDefinitions, and namespaces themselves. A user with broad cluster-scoped permissions can cross namespace boundaries.

  • Namespaces help with: ownership, RBAC scope, network policy scope, quotas, defaults, policy enforcement, and safer operations.
  • Namespaces do not automatically provide: network isolation, node isolation, encryption, strong multi-tenancy, or protection from cluster administrators.

Design a Namespace Strategy

Create namespaces around meaningful ownership and trust boundaries. Common patterns include one namespace per application and environment, such as payments-dev, payments-staging, and payments-prod. Avoid putting unrelated teams in one shared namespace because RBAC, quotas, Secrets, and cleanup become harder to manage safely.

apiVersion: v1
kind: Namespace
metadata:
  name: payments-prod
  labels:
    owner: payments-team
    environment: production
    pod-security.kubernetes.io/enforce: restricted
    pod-security.kubernetes.io/enforce-version: latest
    pod-security.kubernetes.io/audit: restricted
    pod-security.kubernetes.io/warn: restricted

Labels make namespaces easier to inventory and can drive policy. Treat the default, kube-system, kube-public, and kube-node-lease namespaces carefully. Application teams should not deploy ordinary workloads into system namespaces.

Restrict API Access with RBAC

Role-Based Access Control determines who can perform which Kubernetes API actions. A Role grants permissions inside one namespace. A RoleBinding assigns that Role to users, groups, or service accounts.

apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: deployment-operator
  namespace: payments-prod
rules:
  - apiGroups: ["apps"]
    resources: ["deployments"]
    verbs: ["get", "list", "watch", "patch", "update"]
  - apiGroups: [""]
    resources: ["pods", "pods/log"]
    verbs: ["get", "list", "watch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: payments-team-deployment-operator
  namespace: payments-prod
subjects:
  - kind: Group
    name: payments-team
    apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: deployment-operator
  apiGroup: rbac.authorization.k8s.io

Grant the smallest practical set of verbs and resources. Access to create Pods can enable a user to run code under permitted service accounts and mount permitted Secrets. Access to read Secrets directly exposes their contents. Treat both as powerful permissions.

kubectl auth can-i update deployments \
  --namespace payments-prod --as-group payments-team --as alice
kubectl auth can-i list secrets \
  --namespace payments-prod --as-group payments-team --as alice

Avoid casually binding cluster-admin. Use ClusterRoles only when permissions genuinely need to be reusable or cluster-wide, and use RoleBindings to scope reusable ClusterRoles into individual namespaces where possible.

Restrict Network Traffic

Without NetworkPolicy enforcement, Pods can commonly communicate across namespaces. A NetworkPolicy-capable CNI plugin is required; creating policies has no effect if the network implementation does not enforce them.

Start with default deny

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
  namespace: payments-prod
spec:
  podSelector: {}
  policyTypes:
    - Ingress
    - Egress

This selects every Pod in the namespace and denies ingress and egress unless another policy allows it. Roll out default deny carefully: applications often require DNS, monitoring, identity, database, and external API access.

Allow only expected callers

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend-to-api
  namespace: payments-prod
spec:
  podSelector:
    matchLabels:
      app: payments-api
  policyTypes:
    - Ingress
  ingress:
    - from:
        - namespaceSelector:
            matchLabels:
              kubernetes.io/metadata.name: payments-prod
          podSelector:
            matchLabels:
              app: payments-frontend
      ports:
        - protocol: TCP
          port: 8080

In one from item, namespaceSelector and podSelector are combined. Separate list items mean either condition can match, which is broader and a common policy mistake.

Allow DNS egress

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-dns
  namespace: payments-prod
spec:
  podSelector: {}
  policyTypes:
    - Egress
  egress:
    - to:
        - namespaceSelector:
            matchLabels:
              kubernetes.io/metadata.name: kube-system
          podSelector:
            matchLabels:
              k8s-app: kube-dns
      ports:
        - protocol: UDP
          port: 53
        - protocol: TCP
          port: 53

Confirm the actual DNS Pod labels and ports in your cluster before applying this example.

Control Resource Consumption

A ResourceQuota caps aggregate namespace consumption. It helps prevent one team or compromised workload from consuming unbounded cluster resources.

apiVersion: v1
kind: ResourceQuota
metadata:
  name: payments-budget
  namespace: payments-prod
spec:
  hard:
    requests.cpu: "8"
    requests.memory: 16Gi
    limits.cpu: "16"
    limits.memory: 32Gi
    pods: "50"
    services.loadbalancers: "2"
    persistentvolumeclaims: "10"

A LimitRange sets defaults and acceptable boundaries for individual containers or claims. Defaults reduce the chance of workloads being deployed without resource controls.

apiVersion: v1
kind: LimitRange
metadata:
  name: container-defaults
  namespace: payments-prod
spec:
  limits:
    - type: Container
      defaultRequest:
        cpu: 100m
        memory: 128Mi
      default:
        memory: 512Mi
      max:
        cpu: "2"
        memory: 2Gi
kubectl describe resourcequota -n payments-prod
kubectl describe limitrange -n payments-prod

Enforce Pod Security

Pod Security Admission uses namespace labels to enforce Pod Security Standards. The restricted profile blocks many risky settings, including privileged containers and common host access patterns. Use warn and audit first to understand impact before enabling enforce.

kubectl label namespace payments-prod \
  pod-security.kubernetes.io/warn=restricted \
  pod-security.kubernetes.io/audit=restricted \
  --overwrite

System agents may legitimately require broader privileges. Keep them in tightly controlled namespaces and apply the least permissive profile they can support. Admission policies can add organization-specific rules for image registries, required labels, resource settings, or forbidden capabilities.

Operational Guardrails

  • Set a namespace explicitly in manifests and commands instead of relying on the current context.
  • Give each workload a dedicated service account and set automountServiceAccountToken: false when API access is unnecessary.
  • Keep production and non-production in separate namespaces, and use separate clusters when stronger isolation is required.
  • Audit RoleBindings, ClusterRoleBindings, privileged workloads, policy exceptions, and namespace labels regularly.
  • Encrypt Secrets at rest, restrict read access, and use an external secrets system where appropriate.
  • Test NetworkPolicies from allowed and denied sources after every significant change.
  • Use GitOps or reviewed infrastructure code so namespace controls remain reproducible.
  • Alert when quotas approach limits or required policies disappear.

Verify the namespace as a complete control set

kubectl get role,rolebinding,resourcequota,limitrange,networkpolicy \
  -n payments-prod
kubectl get namespace payments-prod --show-labels
kubectl auth can-i --list -n payments-prod --as=<user>

Review the output from the perspective of a compromised workload and a human operator. Ask what each identity can read, create, modify, reach over the network, and consume.

Defense-in-depth model: use namespaces to define ownership, RBAC to control API actions, NetworkPolicy to control traffic, quotas and limits to control consumption, and admission policies to reject unsafe workloads.
