How to Use Kubernetes Deployments and Services to Manage Pods

Pods run your containers, but they are replaceable and their IP addresses change. Deployments keep the right Pods running and roll out new versions. Services give those Pods a stable network endpoint.

Published June 12, 202615 min readKubernetes

Core pattern: clients send traffic to a Service, the Service selects ready Pods by label, and a Deployment continuously reconciles those Pods to the desired replica count and template.

Pods, Deployments, and Services

A Pod is the smallest Kubernetes workload unit. It contains one or more tightly coupled containers that share networking and volumes. Creating standalone Pods is useful for experiments, but it is rarely the right way to run a long-lived stateless application.

A Deployment declares a desired state for replicated Pods. It creates and manages ReplicaSets, replaces failed Pods, scales the application, and performs rolling updates. A Service selects Pods using labels and exposes them through a stable DNS name and virtual IP.

  • Deployment: manages lifecycle, replicas, updates, and rollback.
  • Pod: runs the application containers.
  • Service: provides discovery and load balancing across ready matching Pods.

Build a Complete Application

1. Create a Deployment

This example runs three NGINX Pods. The Deployment selector must match the labels in the Pod template and cannot be changed after creation.

apiVersion: apps/v1
kind: Deployment
metadata:
  name: web
  namespace: default
spec:
  replicas: 3
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 0
      maxSurge: 1
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
        - name: web
          image: nginx:1.27
          ports:
            - name: http
              containerPort: 80
          resources:
            requests:
              cpu: 100m
              memory: 64Mi
            limits:
              memory: 128Mi
          readinessProbe:
            httpGet:
              path: /
              port: http
            initialDelaySeconds: 3
            periodSeconds: 5
          livenessProbe:
            httpGet:
              path: /
              port: http
            initialDelaySeconds: 10
            periodSeconds: 10

2. Create a Service

The Service selector matches app: web. Clients connect to port 80, and the Service forwards traffic to the named container port http.

apiVersion: v1
kind: Service
metadata:
  name: web
  namespace: default
spec:
  type: ClusterIP
  selector:
    app: web
  ports:
    - name: http
      port: 80
      targetPort: http

3. Apply and verify the manifests

kubectl apply -f deployment.yaml
kubectl apply -f service.yaml
kubectl rollout status deployment/web
kubectl get deployment,replicaset,pods,service
kubectl get endpointslices -l kubernetes.io/service-name=web

EndpointSlices should contain the IP addresses of ready Pods. Kubernetes DNS also creates the name web for callers in the same namespace and web.default.svc.cluster.local for the fully qualified name.

4. Test from inside the cluster

kubectl run curl-test --rm -it \
  --image=curlimages/curl --restart=Never -- \
  curl http://web

A ClusterIP Service is reachable only inside the cluster. For external HTTP traffic, place an Ingress or Gateway in front of it. A LoadBalancer Service is another option when a dedicated external load balancer is appropriate.

Scale and Update It

Scale the Deployment

kubectl scale deployment/web --replicas=5
kubectl get pods -l app=web -w

The Service automatically discovers newly ready Pods because they carry the matching label. Removing replicas also requires no Service change.

Roll out a new image

kubectl set image deployment/web web=nginx:1.28
kubectl rollout status deployment/web
kubectl rollout history deployment/web

The Deployment creates a new ReplicaSet and gradually replaces old Pods according to maxSurge and maxUnavailable. Readiness probes prevent traffic from reaching a new Pod before it is ready.

Pause, resume, or roll back

kubectl rollout pause deployment/web
kubectl rollout resume deployment/web
kubectl rollout undo deployment/web
kubectl rollout status deployment/web

Rollback restores a previous Pod template. It does not automatically reverse changes to external dependencies, databases, ConfigMaps, or Secrets, so application releases still need compatibility planning.

Troubleshooting

The Deployment does not reach its desired replicas

kubectl describe deployment web
kubectl get replicaset
kubectl get pods -l app=web -o wide
kubectl describe pod <pod-name>

Look for unschedulable Pods, admission rejection, image pull failures, resource quota limits, missing configuration, and crash loops. Deployment conditions such as Progressing and Available summarize rollout health.

The Service has no endpoints

kubectl get pods --show-labels
kubectl describe service web
kubectl get endpointslices -l kubernetes.io/service-name=web

The Service selector may not match the Pod labels, or every matching Pod may be unready. Compare selectors exactly, including spelling and namespace. Services cannot select Pods across namespaces.

The Service has endpoints, but connections fail

Verify targetPort, the application's listening port, and its bind address. The process must listen on the Pod interface, usually 0.0.0.0, rather than only 127.0.0.1. Then check NetworkPolicies and application logs.

kubectl exec <pod-name> -- curl -v http://localhost:80
kubectl logs <pod-name>
kubectl get networkpolicy

The rollout is stuck

A failing readiness probe, image pull error, insufficient capacity, or overly restrictive rollout strategy can stop progress. Inspect the newest ReplicaSet and Pods before deciding whether to fix forward or roll back.

kubectl rollout status deployment/web
kubectl describe deployment web
kubectl get replicasets --sort-by=.metadata.creationTimestamp

Production Practices

  • Use Deployments for stateless applications; use StatefulSets when stable identity or ordered storage behavior is required.
  • Use immutable image tags or digests so a rollout is reproducible.
  • Set resource requests from measured usage and limits with application behavior in mind.
  • Use readiness probes to control traffic and liveness probes only for recoverable deadlocks.
  • Add a startup probe when slow startup would otherwise trigger liveness failures.
  • Use PodDisruptionBudgets and topology spread constraints for availability during disruptions.
  • Use HorizontalPodAutoscaler when workload demand varies and suitable metrics exist.
  • Keep stable labels for Service selection and separate labels used only for release metadata.

Rule of thumb: manage stateless Pods through a Deployment and send traffic through a Service. Directly managing Pod names or IP addresses creates fragile operations.
