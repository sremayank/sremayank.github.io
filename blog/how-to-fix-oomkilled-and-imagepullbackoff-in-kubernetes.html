How to Fix OOMKilled and ImagePullBackOff in Kubernetes

OOMKilled and ImagePullBackOff are two of the most common Kubernetes pod failures. One happens after a container starts and uses too much memory. The other happens before a container starts because Kubernetes cannot pull the image.

Published June 2, 2026 14 min read Kubernetes

First move: run kubectl describe pod <pod>. The Events section usually tells you whether the failure is a memory limit problem, missing image, registry authentication issue, bad tag, or network problem.

Quick Diagnosis

Start by looking at pod status, container state, previous logs, and events. 

kubectl get pods
kubectl describe pod <pod-name>
kubectl logs <pod-name>
kubectl logs <pod-name> --previous

The two failures look different:

  • OOMKilled: the container started, exceeded its memory limit or was killed under memory pressure, and exited with reason OOMKilled.
  • ImagePullBackOff: the container did not start because the node could not pull the image. Kubernetes retries with increasing delay.

How to Fix OOMKilled

OOMKilled means the container was terminated because memory usage crossed a boundary. Most commonly, the container exceeded its configured memory limit. The exit code is often 137, which means the process received SIGKILL.

1. Confirm the container was OOM killed

kubectl describe pod <pod-name>

Look for output like this under container state or last state: 

Last State:   Terminated
Reason:       OOMKilled
Exit Code:    137

2. Check memory requests and limits

kubectl get pod <pod-name> -o jsonpath='{.spec.containers[*].resources}'

A container with a memory limit that is too low will restart even if the node has free memory. For example, if the application regularly needs 700Mi but the limit is 512Mi, Kubernetes will kill it when it crosses that limit.

3. Check actual memory usage

kubectl top pod <pod-name>
kubectl top pod <pod-name> --containers

If metrics are not available, check your monitoring stack, container logs, runtime metrics, or application profiler. For JVM, Node.js, Python, and Go applications, the runtime may need explicit memory tuning so it understands the container limit.

4. Increase the memory limit carefully

If the workload genuinely needs more memory, increase the request and limit. Do this intentionally: request should represent expected baseline usage, while limit should represent the upper bound the application can safely use. 

resources:
  requests:
    memory: "512Mi"
    cpu: "250m"
  limits:
    memory: "1Gi"
    cpu: "1"

5. Fix leaks or spikes

Raising the limit is not always the real fix. If memory grows continuously until the pod dies, look for memory leaks, unbounded caches, large batch sizes, oversized queues, expensive request payloads, or missing backpressure. If memory spikes only during traffic bursts, consider concurrency limits, streaming, pagination, or smaller worker batch sizes.

6. Roll out and watch restarts

kubectl rollout restart deployment/<deployment-name>
kubectl rollout status deployment/<deployment-name>
kubectl get pods -w

After the rollout, watch restart counts, memory graphs, latency, and application-level errors. The goal is not just to keep the pod alive; the goal is to make memory behavior predictable.

How to Fix ImagePullBackOff

ImagePullBackOff means Kubernetes tried to pull the container image, failed, and is now backing off before trying again. The underlying event usually says why.

1. Read the pod events

kubectl describe pod <pod-name>

Common event messages include manifest unknown, pull access denied, unauthorized, not found, or network timeout errors.

2. Verify image name and tag

Typos are common. Confirm the registry hostname, repository path, image name, and tag. 

kubectl get pod <pod-name> -o jsonpath='{.spec.containers[*].image}'

If your manifest uses my-app:latest, consider switching to immutable version tags or digests. They make rollbacks and audits much clearer.

3. Confirm the image exists in the registry

Make sure the image was actually pushed by CI. A deployment can fail if the manifest references a tag that the build job never published. 

docker pull registry.example.com/team/my-app:1.2.3

4. Fix registry authentication

Private registries require credentials. In Kubernetes, those credentials are usually provided with an imagePullSecrets entry or through cloud-provider node identity. 

kubectl create secret docker-registry regcred \
  --docker-server=registry.example.com \
  --docker-username=<username> \
  --docker-password=<password> \
  --docker-email=<email>
spec:
  imagePullSecrets:
    - name: regcred
  containers:
    - name: app
      image: registry.example.com/team/my-app:1.2.3

5. Check namespace and service account

Secrets are namespace-scoped. If the Deployment runs in production, the pull secret must exist in production. If you attach the secret to a service account, verify the pod is using that service account. 

kubectl get secret regcred -n production
kubectl get serviceaccount default -n production -o yaml

6. Check node network and registry availability

If credentials and image names are correct, look at node-to-registry connectivity. Private clusters, firewall rules, proxy settings, DNS issues, missing NAT, and registry outages can all prevent image pulls.

Prevention Checklist

  • Use CI to verify the image exists before updating Kubernetes manifests.
  • Use immutable image tags or digests instead of relying on latest.
  • Store pull secrets in the same namespace as the workload or attach them to the correct service account.
  • Set memory requests and limits from real usage data, not guesses.
  • Add dashboards for memory usage, restarts, and pod waiting reasons.
  • Alert on repeated OOMKilled, ImagePullBackOff, and CrashLoopBackOff states.
  • Use admission policies to require resource requests, limits, and approved registries.

Operational habit: always capture the event message before changing anything. The event tells you whether you should tune memory, fix code, push an image, correct a tag, or repair registry access.
