From Service VIP to Pod: Tracing Traffic Through Kubernetes

Kubernetes traffic rarely passes through one load balancer. A request may cross a cloud load balancer, an HTTP routing layer, a Service virtual IP, and the cluster network dataplane before it reaches a ready Pod.

Published June 12, 202616 min readKubernetes

Useful mental model: each layer answers a different question. An external load balancer chooses a cluster entry point, Gateway or Ingress chooses an application route, and a Service sends traffic to one of its ready endpoints.

The Traffic Distribution Layers

The phrase load balancing can hide several independent mechanisms. Understanding which layer owns a decision makes network failures much easier to diagnose.

     |
     v
External load balancer
     |
     v
Gateway or Ingress controller
     |
     v
Kubernetes Service virtual IP
     |
     v
Cluster dataplane selects a ready endpoint
     |
     v
Application Pod
  • External load balancer: exposes an address outside the cluster and distributes traffic to cluster entry points.
  • Gateway or Ingress: performs application-aware routing using hosts, paths, headers, or other protocol rules.
  • Service: provides stable discovery and distributes connections across matching ready endpoints.
  • Network dataplane: implements Service forwarding using technologies such as iptables, IPVS, or eBPF, depending on the cluster.

A Kubernetes Service is not necessarily a proxy process that every packet physically passes through. Its ClusterIP is a virtual address. The node networking implementation programs rules that translate or route traffic toward an endpoint.

East-West Traffic Inside the Cluster

East-west traffic moves between workloads inside the cluster. A client normally calls a Service DNS name rather than a Pod IP because Pod addresses change during scaling, rescheduling, and rollouts.

frontend Pod
     |
     | DNS lookup: api.production.svc.cluster.local
     v
Service ClusterIP: 10.96.18.42:80
     |
     | dataplane chooses an EndpointSlice address
     v
api Pod: 10.244.3.17:8080

Services discover endpoints through labels

apiVersion: v1
kind: Service
metadata:
  name: api
  namespace: production
spec:
  selector:
    app: api
  ports:
    - name: http
      port: 80
      targetPort: http

The Service selector identifies Pods with app: api. The control plane publishes eligible addresses in EndpointSlices. Readiness matters: unready Pods are normally removed from serving endpoints so new traffic does not reach an application that cannot safely handle it.

kubectl get service api -n production
kubectl get endpointslices -n production \
  -l kubernetes.io/service-name=api
kubectl get pods -n production -l app=api -o wide

Distribution is usually connection-oriented

Service traffic distribution does not guarantee that every Pod receives exactly the same number of requests. Long-lived HTTP keep-alive, HTTP/2, gRPC, database connections, and WebSockets can keep many requests on one selected backend. If even request distribution matters, use application-aware proxying, tune connection behavior, or design the application to tolerate uneven load.

North-South Traffic from Outside

North-south traffic enters or leaves the cluster. A LoadBalancer Service commonly asks the infrastructure provider to create an external or private load balancer. The exact implementation varies by cloud provider and cluster platform.

apiVersion: v1
kind: Service
metadata:
  name: gateway
  namespace: ingress-system
spec:
  type: LoadBalancer
  selector:
    app: gateway
  ports:
    - name: https
      port: 443
      targetPort: 8443

  v

  +-- shop.example.com --> shop Service --> shop Pods

Gateway API or Ingress lets one routing layer expose multiple HTTP applications. This is often more economical and operationally consistent than provisioning a dedicated cloud load balancer for every application Service.

The exact backend path is implementation-specific. Some Gateway and Ingress controllers send traffic through the Service virtual IP, while others watch EndpointSlices and proxy directly to Pod addresses. Inspect the controller documentation and metrics when the distinction matters.

Service types expose different boundaries

  • ClusterIP: stable address reachable inside the cluster.
  • NodePort: exposes a port on every eligible node, commonly used behind another load balancer.
  • LoadBalancer: requests an infrastructure load balancer and usually builds on Service networking.
  • Headless Service: publishes endpoint addresses through DNS without a ClusterIP, allowing clients to choose endpoints directly.

Traffic Controls and Tradeoffs

Preserve client source IP when required

For a LoadBalancer or NodePort Service, externalTrafficPolicy: Local can preserve the original client source IP by sending external traffic only to endpoints local to the receiving node. The tradeoff is that traffic can become uneven when nodes run different numbers of Pods, and nodes without a local endpoint cannot serve application traffic.

spec:
  type: LoadBalancer
  externalTrafficPolicy: Local

With externalTrafficPolicy: Cluster, traffic can reach endpoints anywhere in the cluster, usually improving distribution but potentially adding another network hop and obscuring the source address.

Use session affinity only when necessary

sessionAffinity: ClientIP attempts to keep a client IP connected to the same endpoint for a period. It can help legacy stateful applications, but it also creates uneven load and does not replace durable shared session storage.

Account for topology

Sending traffic to nearby endpoints can reduce latency and cross-zone cost. Topology-aware routing and traffic policies can influence locality, but they need enough healthy endpoints in each zone. Availability should remain more important than avoiding a cross-zone hop.

Health checks exist at multiple layers

Application readiness controls whether a Pod is an eligible Service endpoint. An external load balancer may separately check nodes or gateway endpoints. Gateway controllers can also perform backend health checks. A failure at one layer can remove traffic even while another layer appears healthy.

A Layer-by-Layer Troubleshooting Workflow

1. Confirm the application is healthy

kubectl get pods -n production -l app=api
kubectl describe pod <pod-name> -n production
kubectl logs <pod-name> -n production
kubectl exec <pod-name> -n production -- curl -v http://localhost:8080/health

If the application does not answer inside its own container, the load-balancing layer is not the first problem to solve.

2. Confirm Service selection and endpoints

kubectl describe service api -n production
kubectl get pods -n production --show-labels
kubectl get endpointslices -n production \
  -l kubernetes.io/service-name=api -o yaml

No endpoints usually means the selector does not match or Pods are not ready. Incorrect endpoint ports usually point to a mismatch between targetPort and the container's listening port.

3. Test Service DNS and ClusterIP

kubectl run network-test -n production --rm -it \
  --image=curlimages/curl --restart=Never -- sh

nslookup api
curl -v http://api/health

If DNS fails, inspect cluster DNS. If DNS succeeds but the Service fails, inspect Service rules, EndpointSlices, NetworkPolicies, and the CNI or Service dataplane.

4. Inspect the external routing layer

kubectl get service -n ingress-system
kubectl get gateway,httproute -A
kubectl get ingress -A
kubectl logs -n ingress-system deploy/<controller-name>

Check the external address, listener configuration, routes, TLS certificates, backend references, controller status, health checks, firewall rules, and provider events.

Production Design Checklist

  • Draw the complete traffic path, including every health check and network boundary.
  • Use readiness probes so broken Pods leave Service endpoints before receiving traffic.
  • Use Gateway API or Ingress for shared HTTP routing and Services for stable backend discovery.
  • Monitor endpoint counts, request distribution, connection duration, retries, and zone balance.
  • Test node, zone, gateway, and backend failures rather than only normal traffic.
  • Document source-IP requirements before choosing external traffic policies.
  • Use NetworkPolicies and TLS controls alongside routing; reachability is not authorization.

When traffic fails: start at the Pod and move outward one layer at a time. This prevents a healthy cloud load balancer from distracting you from an empty EndpointSlice, or a healthy Service from distracting you from a broken Gateway route.
