How to Troubleshoot Unhealthy Kubernetes DaemonSets

DaemonSets run node-level agents such as log collectors, monitoring exporters, network components, and security tools. When one becomes unhealthy, the failure is often tied to a specific node, scheduling rule, host dependency, or rollout.

Published June 12, 202614 min readKubernetes

First move: compare DESIRED, CURRENT, READY, UP-TO-DATE, and AVAILABLE with kubectl get daemonset -n <namespace>. Those numbers tell you whether the problem is scheduling, startup, readiness, or rollout progress.

How DaemonSets Work

A DaemonSet ensures that eligible nodes run a copy of a Pod. Unlike a Deployment, which aims for a replica count, a DaemonSet calculates its desired count from nodes that match its scheduling rules. Adding an eligible node creates another Pod; removing a node removes that need.

This makes DaemonSet health inseparable from node health. A single broken Pod may indicate a host filesystem, kernel, runtime, architecture, taint, capacity, or networking issue on that node rather than a cluster-wide application bug.

Read the status columns

kubectl get daemonset -n kube-system
kubectl get daemonset <name> -n <namespace> -o wide
  • DESIRED: eligible nodes that should run the Pod.
  • CURRENT: nodes currently running or creating the Pod.
  • READY: Pods passing readiness checks.
  • UP-TO-DATE: Pods using the latest Pod template.
  • AVAILABLE: Pods ready for at least the configured minimum ready time.

A Fast Diagnostic Workflow

1. Inspect the DaemonSet and its events

kubectl describe daemonset <name> -n <namespace>
kubectl get events -n <namespace> --sort-by=.lastTimestamp

Check selectors, node selectors, affinity, tolerations, update strategy, image, desired count, and recent events. Events often reveal failed scheduling, image pull errors, admission policy rejection, or rollout problems.

2. List every Pod with its node

kubectl get pods -n <namespace> \
  -l app=<label-value> -o wide
kubectl get pods -n <namespace> \
  -l app=<label-value> --field-selector=status.phase!=Running

Look for patterns. One failing Pod suggests a node-specific issue. Every Pod failing suggests a shared image, configuration, permission, or application problem. Missing Pods suggest scheduling eligibility rather than container startup.

3. Describe and read logs from an unhealthy Pod

kubectl describe pod <pod-name> -n <namespace>
kubectl logs <pod-name> -n <namespace> --all-containers
kubectl logs <pod-name> -n <namespace> --all-containers --previous

The Events section explains scheduling and mount failures. Current logs explain live failures. Previous logs are essential when a container is repeatedly restarting.

4. Compare eligible nodes with scheduled Pods

kubectl get nodes --show-labels
kubectl describe node <node-name>
kubectl get pods -n <namespace> -l app=<label-value> \
  -o custom-columns=POD:.metadata.name,NODE:.spec.nodeName,READY:.status.containerStatuses[*].ready

Check whether labels match the DaemonSet's nodeSelector or affinity, whether taints are tolerated, and whether the node is Ready. Also inspect memory, disk, PID pressure, allocatable resources, and architecture.

Common Failure Patterns

DESIRED is lower than the node count

This can be correct. DaemonSets only target eligible nodes. Review nodeSelector, node affinity, and taints. Control-plane nodes commonly have taints that require explicit tolerations. Do not add broad tolerations until you confirm the agent is safe and necessary on those nodes.

spec:
  template:
    spec:
      nodeSelector:
        kubernetes.io/os: linux
      tolerations:
        - operator: Exists
          effect: NoSchedule

DESIRED is correct, but CURRENT is lower

Pods are not being created or scheduled everywhere. Look for admission policy failures, insufficient CPU or memory, untolerated taints, host port conflicts, and namespace quotas. DaemonSet Pods still need enough node capacity to schedule.

CURRENT is correct, but READY is lower

The Pods reached nodes but did not become ready. Check container logs, readiness probes, mounted configuration, Secrets, service accounts, host paths, and host networking. Node agents often depend on paths such as /var/log or the container runtime socket; a path can differ across node images.

CrashLoopBackOff on one node

Compare that node with healthy nodes. Check its operating system image, kernel, CPU architecture, runtime, mounted paths, available disk, and local ports. A multi-architecture cluster also requires an image manifest supporting every targeted architecture.

kubectl get node <node-name> \
  -o jsonpath='{.status.nodeInfo.osImage}{"\n"}{.status.nodeInfo.architecture}{"\n"}{.status.nodeInfo.containerRuntimeVersion}{"\n"}'
kubectl top node <node-name>

Pending because of a host port conflict

A DaemonSet using hostPort can schedule only if that port is free on the target node. Describe the pending Pod and identify the conflicting workload. Prefer ordinary Pod networking unless the agent truly requires a host port.

ImagePullBackOff or permission failures

Verify the image tag, registry access, pull secret namespace, service account, security context, and required capabilities. Privileged node agents may also be rejected by Pod Security Admission or another admission policy.

Troubleshooting Rollouts

With the default RollingUpdate strategy, Kubernetes replaces DaemonSet Pods gradually. A broken image or readiness probe can stall the rollout and leave a mix of old and new versions.

kubectl rollout status daemonset/<name> -n <namespace>
kubectl rollout history daemonset/<name> -n <namespace>
kubectl get pods -n <namespace> -l app=<label-value> \
  -L controller-revision-hash

Check maxUnavailable, Pod readiness, and whether old Pods can terminate. For a bad release, roll back after confirming the earlier revision is safe:

kubectl rollout undo daemonset/<name> -n <namespace>
kubectl rollout status daemonset/<name> -n <namespace>

Production Prevention Checklist

  • Set realistic resource requests so every targeted node can schedule the agent.
  • Use readiness and liveness probes that test meaningful, local health.
  • Build and verify multi-architecture images when node architectures differ.
  • Keep node selectors and tolerations narrow and documented.
  • Alert when desired, ready, and available counts diverge.
  • Canary changes on a labeled node pool before a cluster-wide rollout.
  • Test host path, kernel, runtime, and permission assumptions against every node image.
  • Use a PodDisruptionBudget only where it matches the operational need; it does not fix unhealthy Pods.

Useful mental model: missing DaemonSet Pods usually point to eligibility or scheduling. Existing but unhealthy Pods usually point to startup, readiness, host dependencies, or permissions.
