How to Connect Kubernetes Pods to Persistent Storage

Container filesystems are temporary. Persistent storage keeps application data beyond a Pod restart or replacement by separating storage lifecycle from Pod lifecycle.

Published June 12, 202616 min readKubernetes

Core pattern: a Pod mounts a PersistentVolumeClaim. The claim requests capacity and access characteristics. Kubernetes binds it to a suitable PersistentVolume, often dynamically created by a StorageClass and CSI driver.

The Kubernetes Storage Model

  • PersistentVolume (PV): a cluster storage resource backed by a disk, filesystem, or storage service.
  • PersistentVolumeClaim (PVC): a namespace-scoped request for storage made by a workload.
  • StorageClass: describes a storage tier and how a CSI driver should provision it.
  • CSI driver: integrates Kubernetes with the underlying storage provider.

With dynamic provisioning, you normally create a PVC and let its StorageClass create the PV. Static provisioning is useful when an administrator has already created a volume that must be matched to a claim.

Understand access modes

  • ReadWriteOnce (RWO): read-write by a single node.
  • ReadOnlyMany (ROX): read-only by many nodes.
  • ReadWriteMany (RWX): read-write by many nodes.
  • ReadWriteOncePod (RWOP): read-write by a single Pod where supported.

Access modes describe attachment capabilities, not a complete application locking system. Support depends on the CSI driver and storage backend. For example, many cloud block disks support RWO but not RWX.

Connect a Pod to Storage

1. Inspect available StorageClasses

kubectl get storageclass
kubectl describe storageclass <storage-class-name>

Identify the default class, provisioner, reclaim policy, volume binding mode, and whether volume expansion is allowed. A class using WaitForFirstConsumer waits until a Pod is scheduled before provisioning storage, which helps choose the correct zone.

2. Create a PersistentVolumeClaim

apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: app-data
spec:
  accessModes:
    - ReadWriteOnce
  storageClassName: standard
  resources:
    requests:
      storage: 10Gi

Omit storageClassName to use the cluster's default StorageClass. Set it to an empty string only when you intentionally want no dynamic StorageClass selection.

3. Mount the claim into a Pod

apiVersion: v1
kind: Pod
metadata:
  name: storage-demo
spec:
  containers:
    - name: app
      image: nginx:1.27
      volumeMounts:
        - name: data
          mountPath: /usr/share/nginx/html
  volumes:
    - name: data
      persistentVolumeClaim:
        claimName: app-data

The volumes section connects the Pod to the claim. The volumeMounts section places that volume inside a particular container. The Pod and PVC must be in the same namespace.

4. Apply and verify

kubectl apply -f pvc.yaml
kubectl get pvc app-data
kubectl apply -f pod.yaml
kubectl get pod storage-demo
kubectl exec storage-demo -- sh -c 'echo persistent-data > /usr/share/nginx/html/index.html'
kubectl delete pod storage-demo
kubectl apply -f pod.yaml
kubectl exec storage-demo -- cat /usr/share/nginx/html/index.html

The recreated Pod mounts the same claim, so the file remains. This demonstrates persistence across Pod replacement, but it is not a substitute for a backup.

Storage for Multiple Replicas

A Deployment with several replicas should not casually mount one RWO claim. Pods may land on different nodes, and applications may corrupt data when sharing a filesystem they were not designed to share. StatefulSets can create one claim per replica using volumeClaimTemplates.

apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: database
spec:
  serviceName: database
  replicas: 3
  selector:
    matchLabels:
      app: database
  template:
    metadata:
      labels:
        app: database
    spec:
      containers:
        - name: database
          image: postgres:17
          env:
            - name: POSTGRES_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: database-credentials
                  key: password
          volumeMounts:
            - name: data
              mountPath: /var/lib/postgresql/data
  volumeClaimTemplates:
    - metadata:
        name: data
      spec:
        accessModes:
          - ReadWriteOnce
        storageClassName: standard
        resources:
          requests:
            storage: 20Gi

Each replica receives a distinct PVC, such as data-database-0. The application is still responsible for replication, consistency, failover, and backups.

Lifecycle and Data Safety

Know the reclaim policy

A dynamically provisioned PV commonly inherits either Delete or Retain from its StorageClass. With Delete, deleting the PVC can delete the underlying storage. With Retain, an administrator must manually recover or clean up the released volume.

kubectl get pv
kubectl get storageclass \
  -o custom-columns=NAME:.metadata.name,RECLAIM:.reclaimPolicy,BINDING:.volumeBindingMode

Expand storage carefully

If the StorageClass allows expansion, increase the PVC request. Confirm that the CSI driver and filesystem support expansion, then watch PVC conditions and application behavior.

kubectl patch pvc app-data \
  -p '{"spec":{"resources":{"requests":{"storage":"20Gi"}}}}'
kubectl get pvc app-data -w

Back up data independently

A PV protects data from Pod replacement. It does not protect against accidental deletion, corruption, compromised credentials, a failed storage region, or a harmful application migration. Use storage snapshots and application-consistent backups, and regularly test restoration.

Troubleshooting

PVC remains Pending

kubectl describe pvc app-data
kubectl get storageclass
kubectl get events --sort-by=.lastTimestamp

Common causes include a missing or incorrect StorageClass, no default class, unsupported access mode, unavailable capacity, CSI driver failure, quota limits, and topology constraints. With WaitForFirstConsumer, Pending before a consuming Pod exists can be expected.

Pod remains Pending or ContainerCreating

kubectl describe pod <pod-name>
kubectl describe pvc app-data
kubectl get volumeattachments

Look for failed mount or attach events, zone mismatch, multi-attach errors, node limits, CSI node plugin health, and permission problems. A block volume still attached to an unavailable node may need time or provider-specific recovery before it can attach elsewhere.

Application receives permission denied

Check the image's runtime user, filesystem ownership, storage backend permissions, and Pod security context. An fsGroup may help supported volume types, but changing ownership on a large filesystem can slow startup.

spec:
  securityContext:
    fsGroup: 2000
    fsGroupChangePolicy: OnRootMismatch

Data disappeared

Confirm that the new Pod mounted the expected PVC and path. Check whether data was accidentally written to the container filesystem, whether a different claim was created, and whether a PVC or PV was deleted. Then inspect snapshots and backups before making further changes.

Operational rule: before deleting a PVC, inspect its bound PV, reclaim policy, snapshots, backups, and restoration plan. PVC deletion can become data deletion.
