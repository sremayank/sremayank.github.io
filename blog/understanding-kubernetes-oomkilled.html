Understanding Kubernetes OOMKilled

OOMKilled is not a random Kubernetes error. It is a signal that a process crossed a memory boundary or the node was under memory pressure. Understanding where that boundary came from is the key to fixing it properly.

Published June 2, 2026 13 min read Kubernetes

Short version: a container is usually marked OOMKilled when it uses more memory than its configured limit. The fix may be a higher limit, but it may also be better application memory management, runtime tuning, smaller batches, or safer autoscaling.

What OOMKilled Means

OOM stands for out of memory. When Kubernetes reports Reason: OOMKilled, the container process was killed because it used too much memory. You will often see exit code 137, which means the process was terminated with SIGKILL.

Kubernetes does not politely ask the process to shut down in this case. The process is killed, the container exits, and the kubelet restarts it if the pod's restart policy allows it. That is why OOMKilled often appears together with increasing restart counts.

Limits, Requests, and QoS

To understand OOMKilled, you need to understand two resource settings:

  • Memory request: how much memory Kubernetes reserves for scheduling decisions.
  • Memory limit: the maximum memory the container is allowed to use.

If a container has a memory limit of 512Mi and tries to use more than that, it can be killed even if the node has unused memory. The limit is enforced at the container level by Linux control groups. 

resources:
  requests:
    memory: "256Mi"
  limits:
    memory: "512Mi"

QoS classes matter during node pressure

Kubernetes assigns pods a Quality of Service class based on requests and limits:

  • Guaranteed: every container has equal CPU and memory request and limit.
  • Burstable: at least one request or limit is set, but the pod is not Guaranteed.
  • BestEffort: no requests or limits are set.

During node memory pressure, BestEffort and Burstable pods are more likely to be evicted than Guaranteed pods. This is separate from a container exceeding its own memory limit, but both are part of the same operational picture: memory must be requested, limited, and monitored with intent.

Common Causes

Memory limit is too low

This is the simplest case. The application needs more memory than the limit allows. It may run fine locally or in staging with low traffic, then die in production under realistic load.

Application memory leak

If memory grows steadily over time and never returns to a healthy baseline, suspect a leak. Common sources include unbounded maps, retained request objects, connection leaks, large in-memory caches, and background jobs that accumulate state.

Runtime does not understand the container limit

Some runtimes need explicit configuration. JVM heap size, Node.js old space size, Python worker concurrency, and Go memory behavior can all affect whether the process respects container limits. The application may believe it has the node's memory available when it only has the container limit.

Large batches or high concurrency

Workers often fail because each message, file, or database result set is loaded into memory. A batch size that looks small during testing can become expensive when concurrency multiplies it.

Sidecars and shared pod pressure

Service mesh proxies, log shippers, and monitoring agents also use memory. If you size only the main application container and ignore sidecars, the pod may be less stable than expected.

How to Diagnose It

1. Confirm the reason and exit code

kubectl describe pod <pod-name>

Look for container state, last state, restart count, reason, and exit code. 

Last State:   Terminated
Reason:       OOMKilled
Exit Code:    137

2. Read previous logs

Current logs may only show the restarted container. Use previous logs to inspect the container that died. 

kubectl logs <pod-name> --previous

3. Check memory usage

kubectl top pod <pod-name>
kubectl top pod <pod-name> --containers

Pair this with historical metrics from Prometheus, Cloud Monitoring, Datadog, New Relic, or your observability platform. The pattern matters: a sudden spike points to traffic or batch behavior, while a steady slope points toward a leak.

4. Compare memory usage to limits

kubectl get pod <pod-name> -o jsonpath='{.spec.containers[*].resources}'

Check whether the limit is realistic for peak usage. Also check whether the request is too low, because a low request can pack too many pods onto the same node and contribute to node pressure.

5. Inspect node pressure and events

kubectl describe node <node-name>
kubectl get events --sort-by=.lastTimestamp

If the node is under memory pressure, you may see eviction events. If only one container repeatedly exceeds its limit, focus on the application and its container resource settings.

How to Prevent It

Set memory based on evidence

Start with measured baseline and peak usage. For example, if the service normally uses 350Mi and peaks near 700Mi during traffic bursts, a 512Mi limit is likely to create avoidable restarts.

Tune the application runtime

For JVM applications, configure heap size with awareness of container memory and leave room for non-heap memory. For Node.js, consider memory flags if the default is not appropriate. For worker processes, tune concurrency and batch size so memory grows predictably.

Use alerts before the kill happens

Alert when memory usage approaches a percentage of the limit, not only after restart counts rise. A useful alert catches the pod at 80-90 percent of its limit before it crosses the boundary.

Make autoscaling memory-aware

Horizontal scaling can help when memory rises with request volume or queue depth. It will not fix a leak by itself, but it can reduce per-pod pressure for traffic-driven workloads. For queue workers, event-driven autoscaling with KEDA can be a strong fit because scaling follows actual backlog.

Right-size sidecars

Give sidecars their own resource requests and limits. A proxy or log agent with no limit can make troubleshooting confusing, while a sidecar with an unrealistically low limit can restart and disrupt the pod.

Use load tests and failure reviews

Run load tests with production-like payload sizes and concurrency. After every OOMKilled event, capture the memory graph, request rate, batch size, container limit, node pressure state, and application logs. Over time, this builds a much clearer sizing model.

Good fix: combine resource tuning with application evidence. A larger memory limit may stop the immediate restart, but the long-term answer comes from understanding why memory reached that point.
