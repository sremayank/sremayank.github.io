Kubernetes Security

Building Security Into Every Layer of a Kubernetes Cluster

Kubernetes security is strongest when the platform continuously verifies identities, rejects unsafe workloads, limits network reachability, protects nodes, and records activity for investigation.

Published June 12, 202618 min readKubernetes

Core idea: no single Kubernetes feature secures a cluster. Build overlapping controls so a compromised credential, vulnerable container, or configuration mistake does not immediately become a cluster-wide incident.

A Cluster-Native Defense Model

Cluster-native security uses Kubernetes identities, API objects, admission flow, workload metadata, and runtime signals as enforcement points. It does not mean Kubernetes is secure automatically. It means the platform gives you context-rich places to define and verify security controls.

Think in terms of prevention, containment, and detection. Prevention blocks unsafe changes. Containment limits blast radius after compromise. Detection shows when controls were bypassed or an authorized action became harmful.

Identity and API Authorization

Authenticate humans and workloads separately

Human access should normally use an external identity provider with short-lived credentials and multi-factor authentication. Workloads should use dedicated service accounts. Avoid sharing credentials between teams, applications, or environments because attribution and revocation become difficult.

Use least-privilege RBAC

RBAC controls Kubernetes API actions. Grant permissions to groups and service accounts rather than individuals where practical, and prefer namespace-scoped Roles over broad ClusterRoles.

apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: application-reader
  namespace: payments
rules:
  - apiGroups: ["apps"]
    resources: ["deployments", "replicasets"]
    verbs: ["get", "list", "watch"]
  - apiGroups: [""]
    resources: ["pods", "pods/log"]
    verbs: ["get", "list", "watch"]

Permissions such as creating Pods, impersonating identities, binding roles, using privileged workload settings, or reading Secrets are especially sensitive. Review them as escalation paths rather than ordinary administrative conveniences.

kubectl auth can-i --list --namespace payments --as=<identity>
kubectl auth can-i create pods --namespace payments --as=<identity>
kubectl auth can-i get secrets --namespace payments --as=<identity>

Reduce service account exposure

Use one service account per workload and disable automatic API token mounting when the application does not call the Kubernetes API.

spec:
  serviceAccountName: payments-api
  automountServiceAccountToken: false

Admission and Workload Hardening

Admission controls inspect API requests after authentication and authorization but before objects are persisted. They can reject unsafe workloads even when an authorized user accidentally submits them.

Enforce Pod Security Standards

Pod Security Admission can enforce a security profile using namespace labels. The restricted profile blocks many high-risk settings. Use audit and warning modes to understand impact before enforcing it broadly.

apiVersion: v1
kind: Namespace
metadata:
  name: payments
  labels:
    pod-security.kubernetes.io/enforce: restricted
    pod-security.kubernetes.io/enforce-version: latest
    pod-security.kubernetes.io/audit: restricted
    pod-security.kubernetes.io/warn: restricted

Harden the container security context

spec:
  containers:
    - name: api
      image: registry.example.com/payments/api@sha256:<digest>
      securityContext:
        allowPrivilegeEscalation: false
        readOnlyRootFilesystem: true
        runAsNonRoot: true
        capabilities:
          drop:
            - ALL
      resources:
        requests:
          cpu: 100m
          memory: 128Mi
        limits:
          memory: 512Mi

Run as non-root, drop capabilities, block privilege escalation, use a read-only root filesystem where possible, and avoid host namespaces, host paths, privileged mode, and unrestricted device access. Resource controls also help contain denial-of-service behavior.

Add policy for organization-specific rules

Use admission policy tooling to require approved registries, image digests, ownership labels, resource requests, safe volume types, and controlled exceptions. Treat exceptions as reviewed, time-bound security decisions.

Network and Data Protection

Move from open networking to explicit flows

NetworkPolicies restrict Pod ingress and egress when enforced by the cluster's CNI plugin. Start with default deny in a test environment, then add policies for documented application flows, DNS, monitoring, identity, and external dependencies.

Internet
   |
   v
Gateway namespace
   |
   | allowed ingress on API port
   v
Application namespace
   |
   | allowed database connection only
   v
Database namespace

All other namespace-to-namespace flows: denied

NetworkPolicy controls reachability, not application authorization. Continue to authenticate callers, authorize actions, and encrypt sensitive traffic.

Protect Secrets and sensitive data

Restrict Secret read access, enable encryption at rest for Kubernetes API data, rotate credentials, and consider an external secrets manager for centralized lifecycle control. Never treat base64 encoding as encryption. Keep sensitive values out of images, manifests, logs, and command-line arguments.

Encrypt communication where trust boundaries require it

Use TLS at external entry points and for sensitive service-to-service communication. A service mesh can help manage workload identity and mutual TLS, but it adds operational complexity and does not replace RBAC, workload hardening, or secure application design.

Image Supply Chain and Runtime

Cluster security begins before an image reaches the cluster. Build from trusted minimal bases, scan dependencies and images, produce software bills of materials, sign artifacts, and verify provenance during admission.

Runtime monitoring
  • Deploy immutable digests instead of mutable tags.
  • Patch base images and dependencies on a defined schedule.
  • Block images with unacceptable risk or unknown provenance.
  • Monitor runtime behavior such as unexpected shells, process execution, file changes, and outbound connections.

Control Plane and Node Protection

Nodes run workload code and hold powerful credentials and network access. A compromised node can affect many Pods, so isolate and minimize the host operating system, patch it promptly, restrict administrative access, and use supported runtime security settings.

  • Limit access to the Kubernetes API endpoint and administrative network paths.
  • Protect control-plane data stores, backups, certificates, and encryption keys.
  • Use separate node pools for workloads with materially different trust or privilege requirements.
  • Avoid ordinary workloads on control-plane nodes.
  • Restrict cloud instance permissions and metadata service access.
  • Keep kubelet, container runtime, CNI, CSI, and node images patched.

Managed Kubernetes reduces some control-plane maintenance work, but customers still own workload identity, authorization, configuration, images, data, and many node-level decisions.

Audit, Detection, and Response

Kubernetes audit logs record API activity and are essential for investigating who changed what and when. Send audit, authentication, admission, node, network, and workload signals to a protected central destination with useful retention.

Alert on meaningful behaviors

  • New ClusterRoleBindings or bindings to powerful roles.
  • Privileged Pods, host mounts, host namespaces, or new policy exceptions.
  • Secret reads outside normal workload or operator behavior.
  • Interactive exec sessions in production Pods.
  • Disabled security policies, audit logging, or monitoring agents.
  • Unexpected image registries, outbound destinations, or runtime processes.

Prepare an incident workflow

Detect --> preserve evidence --> contain identity and workload
       --> identify affected resources --> rotate credentials
       --> rebuild from trusted artifacts --> review control gaps

Practice revoking users, service accounts, certificates, and cloud identities. Know how to isolate workloads without destroying evidence, and verify that cluster and application backups can restore cleanly.

Security review question: if this workload is compromised, what identity can it use, what API objects can it change, what network destinations can it reach, what data can it read, and which controls will detect it?
