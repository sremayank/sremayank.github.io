How to Use Kubernetes Service to Expose Your Application

Pods are temporary. IP addresses change. Deployments scale up and down. A Kubernetes Service gives your application a stable network identity so other workloads and users can reach it reliably.

Published June 2, 2026 12 min read Kubernetes

Quick answer: create a Service that selects your application Pods by label, map a stable Service port to the container port, and choose the Service type based on who needs access: internal workloads, node-level access, cloud load balancer traffic, or direct DNS records.

Why Services Exist

In Kubernetes, a Pod is not a permanent server. It can be rescheduled, replaced during a rollout, evicted from a node, or multiplied when a Deployment scales. Each new Pod gets its own IP address. If another application called the Pod IP directly, that connection would break as soon as the Pod was recreated.

A Service solves this by creating a stable virtual endpoint. Instead of calling Pod IPs directly, clients call the Service name or Service IP. Kubernetes then routes traffic to one of the healthy matching Pods behind it.

The three important pieces

  • Selector: tells the Service which Pods belong behind it, usually using labels like app: web.
  • Port: the port clients use when connecting to the Service.
  • Target port: the port your container actually listens on inside the Pod.

Service Types

Kubernetes gives you several Service types. The right one depends on where the traffic comes from.

ClusterIP

ClusterIP is the default Service type. It exposes the application only inside the cluster. Use it when one workload needs to call another workload, such as an API calling a backend service or a worker calling an internal queue gateway. 

apiVersion: v1
kind: Service
metadata:
  name: web
spec:
  type: ClusterIP
  selector:
    app: web
  ports:
    - port: 80
      targetPort: 8080

NodePort

NodePort exposes the Service on a static port across every node in the cluster. It is useful for simple demos, bare-metal environments, or debugging, but it is usually not the cleanest production entry point because clients must know node addresses and a high port number. 

apiVersion: v1
kind: Service
metadata:
  name: web-nodeport
spec:
  type: NodePort
  selector:
    app: web
  ports:
    - port: 80
      targetPort: 8080
      nodePort: 30080

LoadBalancer

LoadBalancer asks the cloud provider to provision an external load balancer and route traffic to the Service. This is common for internet-facing or private network-facing applications in managed Kubernetes platforms such as GKE, EKS, and AKS. 

apiVersion: v1
kind: Service
metadata:
  name: web-public
spec:
  type: LoadBalancer
  selector:
    app: web
  ports:
    - port: 80
      targetPort: 8080

Headless Service

A headless Service sets clusterIP: None. Instead of giving clients one virtual IP, Kubernetes DNS returns the individual Pod IPs. This is useful for stateful systems that need stable peer discovery, such as databases, brokers, or custom clustering protocols. 

apiVersion: v1
kind: Service
metadata:
  name: database
spec:
  clusterIP: None
  selector:
    app: database
  ports:
    - port: 5432
      targetPort: 5432

Hands-On Example

Suppose you have a simple web application that listens on port 8080. First, create a Deployment with labels. The labels are important because the Service will use them to find the Pods. 

apiVersion: apps/v1
kind: Deployment
metadata:
  name: web
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
        - name: web
          image: nginx:1.27
          ports:
            - containerPort: 80

Now expose it with a Service. In this example, the Service listens on port 80 and sends traffic to the container's port 80. 

apiVersion: v1
kind: Service
metadata:
  name: web
spec:
  selector:
    app: web
  ports:
    - name: http
      port: 80
      targetPort: 80

Apply the manifests: 

kubectl apply -f deployment.yaml
kubectl apply -f service.yaml

Check that the Service exists and has endpoints: 

kubectl get svc web
kubectl get endpoints web
kubectl describe svc web

From another Pod in the same namespace, you can call the Service by name: 

kubectl run curl-test --rm -it --image=curlimages/curl -- sh
curl http://web

Troubleshooting

The Service exists but traffic fails

Start with endpoints. If kubectl get endpoints web shows no addresses, the Service is not selecting any Pods. Check that the Service selector exactly matches the Pod labels. 

kubectl get pods --show-labels
kubectl describe svc web

The Service has endpoints but returns connection refused

This usually means the Service is routing to the Pod, but nothing is listening on the target port. Check the container port, application bind address, and readiness probe. A common mistake is setting targetPort: 8080 when the container listens on 80, or binding the app to 127.0.0.1 instead of 0.0.0.0.

LoadBalancer is stuck in pending

A pending external IP often points to cloud-provider integration, quota, permissions, or unsupported local cluster behavior. In local clusters such as kind or minikube, LoadBalancer may need an extra component like MetalLB or a built-in tunnel command.

Production Tips

  • Use named ports such as http and metrics so manifests stay readable.
  • Keep Services stable and change Deployments behind them during rollouts.
  • Use readiness probes so Services route only to Pods that are ready to receive traffic.
  • Prefer Ingress or Gateway API for many HTTP applications instead of creating one cloud load balancer per Service.
  • Use NetworkPolicies when you need to restrict which Pods can call the Service.
  • Document whether a Service is internal-only, private load balanced, or public internet-facing.

Rule of thumb: use ClusterIP for internal app-to-app traffic, LoadBalancer for simple external exposure, Ingress or Gateway API for HTTP routing, and headless Services for direct Pod discovery.
