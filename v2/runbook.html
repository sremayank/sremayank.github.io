RB-001
Scaling under demand

Queue pressure is a signal, not a capacity plan.

01 / Trigger

Queue depth rises

Consumer lag threatens the response objective.

02 / Diagnose

Confirm demand

Separate a demand spike from stuck workers or delayed metrics.

03 / Decide

Apply policy

Use tenant limits and max replicas to contain the response.

04 / Automate

KEDA scales

Scale workers from queue depth and external signals.

05 / Verify

Watch recovery

Confirm lag, errors, and resource pressure return to normal.

02 / Operating principles

What makes a platform usable at 03:00.

A / Visibility

Context travels with the alert.

SlackOps and useful event data reduce time spent finding the system that is actually failing.

B / Guardrails

Unsafe changes fail before production.

Terraform validation, OPA, secret scanning, and reviewable delivery paths create useful friction.

C / Recovery

Rollback is a designed capability.

Blue-green upgrades and GitOps delivery keep a known-safe path available when the change does not behave.
